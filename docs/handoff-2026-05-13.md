### B. PPT 디자인 적용 (큰 챕터)
디자인 톤 모달 → 생성 → 디자인 입혀진 .pptx 다운로드. 사용자의 `presentation-design-system.md` (Nike editorial minimalism) 기반.

긴 시행착오 흐름 (디버깅 순서):
1. 풀스크린 배경 박스로 `slide.background` API 우회
2. 폰트 슬롯 3종(latin/ea/cs) 후처리
3. self-closing `<a:rPr ... />` 케이스 추가
4. theme1.xml 후처리에서 기존 `<a:font script>` 보존
5. chapter marker 영문 단일화 ('Proof / 성과' → 'Proof')
6. **typeface="Inter" 강제 교체** ← 진짜 원인 발견 (pptxgenjs가 Inter 박아넣음)
7. **Pretendard 폰트 임베드** — `public/fonts/Pretendard-Regular.otf` + `Pretendard-Bold.otf` (각 1.5MB)

최종 후처리 3단계:
- (a) 모든 `<a:latin/ea/cs typeface="X"/>` 의 X를 우리 폰트로 강제 교체
- (b) self-closing rPr → 열고 닫는 태그 + 3종 자식 추가
- (c) latin 자체가 없는 rPr → 3종 자식 추가

### C. 슝 대용량 발송 (마지막 챕터)
사용자가 production에서 2만명 발송 시도 → 504 Gateway Timeout.

원인 진단:
- Vercel maxDuration 300초 한도 초과
- 슝 단건 API(`api.shoong.kr/send`) 한 명씩 호출 × concurrency 5 → 17~30분 소요
- 우리가 만든 에러 로그에도 안 잡힘 (timeout은 catch 블록 안 들어옴)

수정:
- `logError` 적용 → DB 기록 + 사용자 친화 메시지
- `maxDuration` 60 → 300
- `safeFetchJson` 헬퍼 (Dashboard.js) — JSON 파싱 실패 시 친절한 메시지
- **청크 분할 구현**: `chunkOffset` / `chunkSize` 파라미터. 클라이언트가 500명씩 N번 호출.
- 진행률 UI (`shoongBulkProgress` state)

## 3. 다음 작업 — ★ 슝 대량 API 전환 (가장 큰 임팩트)

### 핵심 발견
사용자가 슝 어드민 캡쳐로 발견:
- 발송 이력 관리에 **"대량"** 탭과 **"단건"** 탭이 따로 있음
- 현재 우리는 **단건 API**로 한 명씩 호출 중 → 그래서 매우 느림
- 슝 어드민의 **대량 API**를 쓰면 한 번에 수천 명 처리 가능

### 다음 세션에서 할 일

**1. 슝 대량 발송 API 문서 확인**
- 슝 어드민 좌측 사이드바의 **"개발자 도구"** 또는 **"Shoong 매뉴얼"** 메뉴
- 대량 발송 API의:
  - Endpoint URL (예: `api.shoong.kr/send-bulk` 또는 `/batch` 등 추정)
  - Payload 형식 (대량 명단 묶음 형식)
  - 응답 형식 (개별 성공/실패 정리)

**2. 코드 변경**
- `app/api/tools/shoong-bulk/send/route.js`:
  - `SHOONG_ENDPOINT`를 대량 API URL로 변경 (또는 단건/대량 분기)
  - `runWithConcurrency` 한 명씩 호출 패턴 → 대량 payload 한 번 전송으로
  - 응답 파싱 로직 수정
- `components/Dashboard.js`:
  - 청크 크기 500 → 5000 또는 그 이상으로 늘리기
  - 진행률 UI는 그대로 (청크 수만 줄어들 뿐)

**3. 기대 효과**
- 2만명 발송 시간: 25~30분 → **3~5분**
- 슝 어드민의 "대량" 탭에 표시
- API 호출 횟수 1/100 (요금 절감 가능성)

### 단건 vs 대량 결정 기준
- 작은 명단(예: 100명 미만)은 단건 그대로 유지
- 100명 이상은 대량 API
- 또는 모두 대량 API로 통일 (간단함)
